# TDC Matchmaker Dashboard — Submission Write-up

## Tech Choices

For the frontend I chose Next.js 14 with the App Router, TypeScript, and Tailwind CSS. The App Router's nested layouts made it clean to implement the auth guard at the dashboard layout level — unauthenticated users are redirected to login before any dashboard route renders. The backend is FastAPI with Pydantic for schema validation, `python-jose` for JWT auth, and `passlib` for password hashing. I chose FastAPI over Node.js because the matching algorithm is written in Python and keeping it server-side made more sense than porting the logic to JS. Data is stored as static JSON files (customers and a pool of 120 profiles) — no database was needed for this scope, and it keeps the setup dependency-free. The whole system is deployed on Vercel (frontend) and Render (backend).

## Matching Logic

The algorithm runs a two-phase pipeline. First, profiles are filtered to the opposite gender and run through a weighted scoring function (0–100) with gender-specific logic. For male customers, the system applies hard filters — any woman who is older or taller than the customer is immediately excluded. Remaining candidates are scored on age gap (20pts), income differential (15pts), kids compatibility (20pts), religion (10pts), relocation alignment (10pts), height gap (10pts), diet (5pts), family values (5pts), and marital status (5pts). For female customers, there are no hard filters — the logic is holistic, scoring on age range (15pts), profession tier compatibility (15pts), relocation stance (15pts), kids alignment (15pts), family values (15pts), education parity (10pts), lifestyle compatibility (10pts), and religion (5pts). This distinction was intentional: the male criteria reflect traditional Indian matrimonial expectations as specified, while the female criteria prioritise life compatibility and shared values based on research into how women navigate matrimonial platforms. The top 15 candidates by score are surfaced to the matchmaker.

## How AI Is Used

After the rule-based scoring produces a ranked list, each of the top 15 matches is passed to Groq (Llama 3.3-70b) with a structured prompt containing both profiles and the key matching reasons. Groq returns a short label ("Excellent Match", "High Potential Match", etc.) and a one-sentence warm explanation specific to that pairing — for example, *"Shared TCS background and similar professional ambitions create a strong foundation."* This two-layer approach means the scoring is deterministic and explainable (the reason chips under each card), while the AI adds the human warmth a matchmaker would actually communicate to a client. If the Groq call fails for any reason, the system falls back gracefully to a rule-based label derived from the score range.

## Assumptions Made

The matchmaker (TDC staff) is the primary user of this dashboard — clients don't log in directly. The matching pool and customer profiles are seeded as static JSON with realistic Indian matrimonial data (religion, caste, family values, manglik status, diet, income in LPA). Income is treated as annual LPA throughout. The "Send Match" feature generates a formatted email preview but does not actually send an email — in production this would integrate with an email service. Caste compatibility was intentionally excluded from the scoring weights; religion and family values were used as proxies for cultural compatibility instead, which is a more defensible signal. The female matching logic deliberately avoids hard filters on height and income because those constraints, while common on some platforms, aren't reflective of what most women actually prioritise in a partner.
